webpack入门篇（二）：WHAT IS WEBPACK

webpack是模块管理器(module bundler)

webpack将具有依赖关系的模块生成对应的静态资源

为什么需要新的模块管理器

目前存在的模块管理器不能够很好地满足大的应用（大的单页面应用）。开发一个新的模块管理器的最主要原因是实现代码的拆分和静态资源应该通过模块化无缝地连接到一起。

我尝试扩展现有的模块管理器，但是不能实现所有的目标。

目标

  1. 将依赖树拆分成块，按需加载
  2. 减少初始化的时间
  3. 每个静态资源都能够独立成为一个模块
  4. 能够将第三方库作为模块整合在一起
  5. 能够自定义模块管理器的几乎所有部分
  6. 适合大的工程

webpack有什么不同

代码的拆分

在webpack的依赖树中有两种依赖方式：同步和异步。异步依赖作为分割点，形成一个新的块。对块树进行优化之后，每一个块都生成一个文件。

加载器

webpack只能够处理原生的javascript，但是加载器可以将其他资源转化成javascript。这样，每一个资源组成一个模块。

智能编译

webpack拥有智能编译器可以处理几乎所有的第三方库，在依赖中可以使用表达式，例如：

webpack能够处理最常用的模块处理风格，CommonJs和AMD

插件系统

webpack拥有功能丰富的插件系统，大多数的内部功能都基于插件系统。webpack允许根据自己的需要自定义插件并将共通的插件作为开发源码进行发布。

webpack入门篇（一）：动机

现在的网站已经演变成了各种web应用，有以下特点：

  1. 页面中存在越来越多的javascript

  2. 在现代浏览器中我们可以做更多的事情，更多的效果

  3. 越来越少的页面刷新（越来越多的异步刷新）–导致产生更多的javascript代码

这些特点导致有很多的代码运行在客户端（浏览器）

一个巨大的代码库需要被组织起来。模块系统提供了一种选择，可以将代码库划分成模块

模块系统的风格

目前有多种标准来定义依赖和输出

  1. 标签风格（没有模块化）

  2. CommonJS

  3. AMD和一些方言(some dialects of it)

  4. ES6 module

  5. 更多….

标签风格

这是没有使用模块系统的时候处理代码库模块化的一种方式

模块通过全局对象来输出接口，例如：window对象。模块可以通过全局对象访问到依赖的接口

常见问题：

  1. 全局对象的冲突

  2. 加载的顺序很重要

  3. 开发者必须处理维护模块或者库的依赖关系

  4. 在大的系统中，代码文件比较多且文件之间的依赖关系难以维护

CommonJs: synchronous require(同步require)

这种方式使用的是同步的require方法实现依赖关系和返回输出接口，模块可以通过给exports对象加属性或者给module.exports设置值来指定输出。

这种方式在node.js的服务端使用

优点：

  1. 服务端模块可以重用

  2. 已经有很多可用的模块使用这种方式，可以直接使用（npm）

  3. 使用起来简单而且简洁

缺点：

  1. 阻塞调用不适用于网络，网络请求是异步的

  2. 无法并行请求多个模块

实现：

  • node.js - server-side
  • browserify
  • modules-webmake - compile to one bundle
  • wreq - client-side

AMD: asynchronous require(异步require)

异步模块定义

针对同步模块系统在浏览器中遇到的问题提供的一种异步解决方案

优点：

  1. 符合网络的异步请求方式

  2. 可以并行加载多个模块

缺点：

  1. 代码开销比较大，难于阅读和编写 ？？为什么作者好像对AMD方式比较不认可的样子

  2. 似乎是某一种解决方案

实现：

  • require.js - client-side
  • curl - client-side

ES6 modules

EcmaScript6 增加了一些语法结构，形成了另外一种模块系统

优点：

  1. 静态分析比较容易（Static analysis is easy）

  2. ES的未来标准

缺点：

  1. 浏览器的支持还需要时间

  2. 使用这个方式编写的组件还很少

合理的解决方案

给开发者选择模块风格，保证现在的代码可用，更方便的添加自定义的模块风格

TRANSFERRING（传输）

由于模块需要在客户端使用，所以需要从服务端请求到客户端

目前有两种极端的传输的方式

  1. 一个请求对应一个模块
  2. 一个请求返回所有模块

目前两种方式都被广泛地应用，但都不是最优的处理方案

第一种方式

优点： 按需加载模块

缺点：

  1. 请求过多，增加网络开销

  2. 请求过多，延迟页面的加载

第二种方式

优点： 较少的网络开销，较少的页面延迟

缺点： 没有被require的模块也会被加载

Chunked transferring（分块传输）

一个更灵活的传输方式会更好，对于多数的情况下，采取两种极端方式的折中方案会更好

->当编译所有的模块：将一组模块拆分成多个更小的块

我们拆分成多个更小的请求，包含一组模块的块按需加载，初始的请求不包含所有的代码库，而是较小的

分割点是开发者可以配置的

->一个大的代码库是可能的

上面的写法是比较直接的，可以完成这个功能，但是8个基本相同的结构，你是不是觉得可以用循环呢？所以我们来改一下

由于属性名是不确定的，没有办法使用\$company.self\$i这种形式，所以我们可以使用evaluate来动态的解析属性值

使用evaluate命令有几个注意方面

  1. 代码#evaluate(“\$company.self\$i”) 第一个\$后面不能使用感叹号（静默模式），因为使用静默模式的时候它会先解析\$!company.self这一部分，发现这个部分是null，则使用静默模式处理成了空字符串，继续解析第二个$符号对应的变量，最后的结果是变成了1,2,3..这样的数据

  2. 代码#evaluate(“\$company.self\$i”)，必须使用双引号包含，不然会解析不成功，最后显示成\$company.self1,\$company.self2…

  3. 如果内部使用的是双引号，可以在外层使用单引号。例如#evaluate(‘\$!cols.get(“self1”).columnName’)，可以正常解析

  4. 如果\$company.self1不存在或者为null的时候，会显示成\$company.self1,\$company.self2…

IE8自动开启兼容性视图

有的时候在IE8下打开浏览器访问页面的时候会莫名其妙地跳转到IE8兼容性视图下。

我到目前为止遇到过两次这个事情，留个记录作为以后解决问题的参考。


场景：页面中有一个span（行内元素）嵌套了一个div（块级元素）

原因： 结构嵌套问题

解决方案：合理的页面嵌套。不要在行内元素中嵌套块级元素，如果非要这样使用，设置行内元素的display属性为inline-block


场景：我们有一个树结构，我们给容器加了样式overflow-y: scroll，当树展开的时候，出现横向滚动条的时候，IE8切换到兼容模式

原因： 样式错误

解决方案： 将容器的样式设置为overflow: auto，怀疑可能是因为没有设置x轴的处理方式，当x轴出现滚动条时候出现异常

3. html结构
<ul class="ztree ztree-wrap" id="departmentTree">
</ul>

  • 由于默认的树结构是用li和ul配合生成的，所以页面中用ul作为容器

  • id是必须的，因为ztree用id来识别一棵树

4. 初始化树

我们使用的是简单的数据结构

var settings = {
  data: {
    key: {
      name: "deptName"  // 树节点显示名称对应的key
    },
    simpleData: {
      enable: true,       // 启用简单的数据结构
      idKey: "deptId",    // 节点key
      pIdKey: "deptPId",  // 父节点key
      rootPId: 0          // 根节点父key的值
    }
  }
}

var departments = [
  {deptName: xx公司, deptId: -1, deptPId: 0},
  {deptName: 研发部, deptId: 1, deptPId: -1},
  {deptName: 行政部, deptId: 2, deptPId: -1}
]

$.fn.zTree.init($('#departmentTree'), settings, departments)
5. 展开节点

节点展开处理

var treeObj = $.fn.zTree.getZTreeObj('departmentTree');
var curNode = treeObj.getNodeByParam('deptId', 1);
treeObj.expandNode(curNode, true);

这样处理有一个问题，如果当前节点是叶子节点的时候，展开功能不能focus到当前节点

我们使用下面的处理方式解决当前问题

var treeObj = $.fn.zTree.getZTreeObj('departmentTree');
var curNode = treeObj.getNodeByParam('deptId', 1);
if (curNode.isParent) {
    var currId = curNode.tId;
    curNode = curNode.getParentNode();
    treeObj.expandNode(curNode, true);
    $('#' + currId).focus();
}
else {
    treeObj.expandNode(curNode, true);
}
